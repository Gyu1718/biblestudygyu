# 콘텐츠 규격 — 성서 연구 서고

이 문서는 **연구 문서 자체의 형식과 품질 기준**을 규정한다. `ARCHITECTURE.md`가 사이트의 뼈대(폴더·데이터·기능 레이어)를 다룬다면, 이 문서는 그 안에 담기는 **심층 연구·원어 파싱·인터라이너 문서를 어떻게 쓰는가**를 고정한다.

새 채팅이나 다른 AI로 작업을 이어갈 때, **이 문서를 규격으로 제시하면 결과물의 형식과 품질이 일정하게 유지된다.** 매번 바뀌는 것(성경 본문·주석 자료)과 항상 같아야 하는 것(형식·표기·원칙)을 분리하는 것이 핵심이다.

---

## 0. 대전제 — 집필 역할의 경계

**창작은 연구자(사용자)의 영역이고, AI는 자료·사실의 정리·조합·검증에 한정한다.**

- AI가 하는 일: 주석 대조, 원문–번역 검증, 어휘·어근 계보 추적, 역사 정리, 출처 간 공통점·차이 정리, 형식에 맞춘 조판.
- AI가 넘보지 않는 일: 논지 세우기, 문장 쓰기(문체 창작), 구성 결정, 은유·이미지 선택.
- 예외: 사용자가 시간 절약이나 빠른 초고를 위해 **명시적으로** 맡기는 경우에 한한다.

파싱·인터라이너·관주 정리는 이 "검증·정리" 범주에 속하므로 AI 작업이 적절하다. 반면 심층 연구의 **신학적 논지와 문장**은 사용자 몫이며, AI는 주석 근거의 정리·대조까지만 돕는다.

## 1. 정직성 원칙 (타협 불가)

품질의 토대는 정직성이다. 다음을 반드시 지킨다.

1. **불확실하면 밝힌다.** 원어 형태·모음 표기가 확실치 않으면 추정임을 명시하고, 실제 도구로 재확인하도록 안내한다.
2. **자료 손상을 숨기지 않는다.** (실제 사례: 업로드된 BHS EPUB가 OCR 47%로 손상 → 원문 소스로 못 씀을 고지하고, 마소라 본문 지식으로 작성하되 각주에 재확인 권고.)
3. **AI 지식의 한계를 표시한다.** 특히 **명단 장의 고유명사 모음(vocalization)**은 기억 재구성 오류 위험이 크다. 서사·기도 장은 본문이 확립돼 정확히 다룰 수 있으나, 낯선 인명·지명은 실제 소프트웨어 대조가 필요함을 밝힌다.
4. **원어는 반드시 재확인 경로를 남긴다.** 모든 파싱·인터라이너 문서 각주에: "정밀 인용·논문 작업 시 실제 BHS 및 형태분석 도구(Accordance, Logos, STEP Bible 등)로 재확인할 것."

> **어느 AI로 작업하든 원어 정확성은 모델마다 편차가 크다. 원어는 소프트웨어로 재확인하는 절차를 항상 규격에 포함한다.**

## 2. 번역 본문 대조 규칙

- **개역개정 4판**을 기준으로 한다.
- 인용 구절은 **단어 하나까지** 첨부된 성경 파일과 대조한다. 개역한글·다른 판의 표현이 섞이지 않게 한다.
- 불일치를 발견하면 **첨부 파일 기준으로 수정**한다. (실제 수정 사례: "싸워"→"치고", "힘이 쇠하였으므로"→"힘이 다 빠졌으니", "각 민족의"→"각 족속의".)
- 번역이 원문에 없는 내용을 보충한 지점(예: 느 1:1 "아닥사스다 왕"은 히브리어에 없이 번역이 보충)은 인터라이너에서 짚어 준다.
- 검증은 스크립트로 한다: 인용문에서 공백을 제거해 원문과 비교, 생략부호(…) 포함 인용은 스킵.

## 3. 주석 표기 (다섯 주석 칩)

느헤미야 기준. 책이 바뀌면 해당 책의 표준 주석으로 교체하되, **약칭 한 글자 + 색상 칩** 방식은 유지한다.

| 칩 | 주석가 | 판본 | 해석 축 |
|---|---|---|---|
| **K** | Kidner | TOTC 1979 | 경건·수사·목회적 적용 |
| **F** | Fensham | NICOT 1982 | 양식·어휘·본문비평 |
| **T** | Throntveit | Interpretation 1992 | 구조=신학(교차구조·인클루지오·코다) |
| **B** | Bolin | NCBC 2012 | 자료비평·제국사·사회학 |
| **H** | Harrington | NICOT 2022 | 제2성전기·수용사 |

칩 색상(CSS 변수): `--k`#7a5a38 · `--f`#37665a · `--t`#3c5a83 · `--b`#655a8c · `--h`#8a4658.
HTML: `<span class="c h">H</span>` 형태. 노트 안에서 어느 주석의 관찰인지 칩으로 표시한다.

---

## 4. 원어 표기 규칙

- 원어는 **(발음, 뜻)을 병기**한다. 예: <span dir="rtl">דָּרַשׁ</span>(다라쉬, 찾다·구하다).
- 발음은 **한글 음역**을 기본으로 한다. 인터라이너 표에서는 학술 로마자 음역을 함께 제공해 문법 구조(접두·접미)가 드러나게 한다.
- 어근은 **세 자음 어근**으로 표기하고, 파생 낱말을 함께 든다. 예: 어근 <span dir="rtl">שׁמר</span>(š-m-r) — 명사 <span dir="rtl">מִשְׁמָר</span>(미쉬마르, 당번).
- 히브리어는 RTL(`direction:rtl`), Noto Serif Hebrew. 그리스어는 Noto Serif, 라틴어도 세리프.

## 5. 문장 작성 규칙 (한국어 산문, 항상 적용)

심층 연구의 산문과 모든 노트·해설에 적용한다.

1. **부정 후 긍정 구조 금지**: "~이 아닙니다", "~가 아니라", "~이기 전에", "A가 아니라 B입니다" 형태를 쓰지 않는다.
2. **대조 뒤집기 구조 금지**: 두 개념을 세워 방향을 뒤집는 구성, "그러나/오히려/다만" 회피.
3. **AI 상투어 금지**: "깊은", "바라보다", "자리" 같은 표현.
4. **논문식 전환어 금지**: "현대 사회에서는", "결론적으로".
5. **설교조 처방 금지**: "~해야 합니다" 식.
6. **주술 호응**을 정확히 지킨다.
7. **조사**를 어법에 맞게 정확히 쓴다.

## 6. 문서별 형식 규격

세 문서는 **각각 다른 층위**를 담당한다. 한 파일로 융합하지 않고, 절 앵커(`id="vN"`)로 상호 연결한다(그래야 데이터 유실이 없다).

### 6-1. 심층 연구 (`ot/<book>/chNN.html`)

- **목적**: 다섯 주석을 절 범위 단위로 종합한 본격 주해.
- **구조**: 좌측 목차(sticky) + 본문. 섹션 순서 = STRUCTURE(장 구조 개관) → 절 범위별 주해(eyebrow에 "1:1–3" 식 표기) → CROSS-REFERENCES(학술 관주 총람) → SYNTHESIS(장 신학 종합).
- **절 범위별 주해**: 히브리어 표제 + 음역·뜻(gloss) + 주해 본문 + 주석 칩 노트.
- **CROSS-REFERENCES 섹션**: 엄선된 학술 관주(성경 내 + 고대 자료 + 2차 문헌)에 주석 근거 칩을 붙인 표. **이것은 openbible 관주와 별개로 유지한다**(질적 관주). 예: "1:6 왕상 8:29 — 솔로몬 봉헌 기도의 어법 T H".
- **분량**: 장당 실질 주해. 절 앵커는 `id="v23"`(2–3절 묶음)처럼 범위를 반영.
- **문체**: 위 §5 문장 규칙을 지키는 산문. 논지·문장은 사용자 창작 영역.

### 6-2. 원어 파싱 (`ot/<book>/parsing/chNN.html`)

- **목적**: 장의 **핵심어를 선별**해 발음·뜻·문법·어근으로 파싱.
- **공통 CSS**: `parsing/parsing.css` 한 파일을 13편이 공유(스타일 수정은 이 파일만).
- **카드 구조** (`.lex`):
  ```html
  <div class="lex">
    <div class="top">
      <span class="word">히브리어</span>
      <span class="translit">한글음역</span>
      <span class="gloss">뜻</span>
      <span class="ref">느 N:M</span>
    </div>
    <dl>
      <dt class="p">문법</dt><dd><span class="pp">품사·어간·인칭·성·수·상태</span> — 설명.</dd>
      <dt class="root">어근</dt><dd><span class="hb">어근</span>(자음) — 파생 낱말.</dd>
    </dl>
    <div class="note"><span class="c h">H</span> 주석 근거 관찰.</div>
  </div>
  ```
- **개역개정 구절**(`.krv`)을 절마다 병기하고 전수 대조한다.
- **카드 밀도**: 1장 기준(19카드)으로 조밀하게. 명단 장은 대표어 위주(7카드 수준).
- **표제 히브리어**: 각 장의 주제를 담는 히브리어 구절을 hero에 둔다(예: 1장 <span dir="rtl">זְכָר־נָא</span>).

### 6-3. 인터라이너 (`ot/<book>/interlinear/chNN.html`)

- **목적**: 한 절의 **모든 낱말**을 축자 대역하고 전수 문법 분석.
- **공통 CSS**: `interlinear/interlinear.css`.
- **절마다 3층 구조**:
  1. **개역개정 head**: `<div class="verse-head"><span class="vnum">1:1</span><span class="vkrv">본문</span></div>`
  2. **축자 대역 띠** (RTL): 각 낱말을 `<span class="iw"><span class="ih">히브리어</span><span class="it">음역</span><span class="ig">뜻</span></span>`로. 오른쪽→왼쪽으로 읽힌다.
  3. **단어 분석 표** (7열): `# / 히브리어 / 음역(한글+로마자) / 문법 분석 / 표제어·어근 / 뜻 / 문장 기능`. 셀 클래스: `w-num` `w-heb` `w-tr`(`.ko` 포함) `w-parse` `w-lex`(`.lm`표제어/`.rt`어근/`.rt-g`뜻) `w-gloss` `w-fn`.
  4. **구문 관찰** (`.vnote`): 절의 구문·번역 관찰에 주석 칩.
- **모든 절**(v1..vN)에 앵커. 표는 좁은 화면에서 좌우 스크롤.
- **각주**: BHS·형태분석 도구 재확인 권고(§1).

## 7. 관주 (cross-references)

- **두 종류를 모두 유지한다.**
  - **학술 관주**: 심층 연구의 CROSS-REFERENCES 섹션. 주석 근거가 붙은 엄선 관주(질적).
  - **openbible 관주**: `data/xrefs/<book>.json`. 대중 투표 기반 방대한 목록(양적, OpenBible.info, CC-BY). 절별 자동 표시, 투표수 순 정렬, 상위 8개 우선 + 더보기.
- 관주는 **참고용 연결**이며, 신학적 해석은 심층 연구를 따른다.
- 생성: `python3 tools/build_xrefs.py cross_references.txt <BookCode>`.

## 8. 시각 규격 (색·서체·테마)

- **팔레트**: 청람 `--lapis`#1f4e78 + 황토 `--ochre`#a8823c + 양피지 `--paper`#fbfaf6. 어근 `--root`, 문법 `--parse`.
- **서체**: 본문 Noto Serif KR, 캡션·데이터 Noto Sans KR, 히브리어 Noto Serif Hebrew.
- **색은 CSS 변수로만** 쓴다. 하드코딩 금지. `assets/theme.css`가 라이트/다크 두 벌을 정의하므로, 변수만 쓰면 다크 모드가 자동 적용된다.
- **다크/라이트·반응형**: `data-theme="auto|light|dark"`, 중단점 모바일600/태블릿900/와이드1180. 접근성(키보드 포커스·`prefers-reduced-motion`) 준수.
- **구조 장치는 내용을 인코딩**해야 한다. 절 범위 eyebrow, 주석 칩, 어근 색 등은 장식이 아니라 정보다.

## 9. 검증 (제출 전 필수)

새 문서를 만들면 반드시 통과시킨다.

1. **태그 균형**: 여닫는 태그 수 일치(div/section/table/tr/td/span/a 등).
2. **앵커 무결성**: `href="#v.."`가 모두 실재하는 `id`를 가리킨다.
3. **개역개정 전수 대조**: 인용 구절이 첨부 파일과 완전 일치(불일치 0).
4. **관주·이동 지도 재생성**: `build_xrefs.py`, `build_links.py` 실행.
5. **사이트 검사**: `python3 tools/validate.py` → 오류 0.
6. **공백 태그 없음**: `</div >` 같은 것 금지.
7. **인터라이너 표 열 수** 일치, **파싱 카드 필드** 누락 없음.

## 10. 이 규격을 다른 AI/채팅에 넘기는 법

`HANDOFF.md`에 첫 메시지 예시 문구가 있다. 핵심은:

1. 이 패키지(zip)와 `ARCHITECTURE.md`·`CONTENT-SPEC.md`를 함께 준다.
2. "이 두 문서를 규격으로 삼아, 형식·표기·원칙을 그대로 지켜라"고 지시한다.
3. 이번 작업 대상(책·장, 어느 문서 종류)과 자료(성경 파일·주석)를 첨부한다.
4. 빈 템플릿(`templates/`)의 칸을 채우는 방식으로 작업하게 한다.
5. **원어는 소프트웨어로 재확인**하는 절차를 잊지 않도록 상기시킨다.
