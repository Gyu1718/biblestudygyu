# 인계 가이드 — 다른 채팅·다른 AI로 작업 이어가기

이 프로젝트를 새 채팅이나 다른 AI로 이어갈 때, **결과물의 형식과 품질을 일정하게 유지**하는 방법을 정리한다. 핵심 원리는 하나다.

> **매번 바뀌는 것**(성경 본문·주석 자료)과 **항상 같아야 하는 것**(형식·표기·원칙·틀)을 분리한다. 후자를 문서·CSS·템플릿으로 고정하고, 그것을 규격으로 제시한다.

---

## 무엇을 넘기는가

새 작업을 맡길 때 아래 세 가지를 함께 준다.

1. **이 패키지 전체(zip)** — 특히 다음이 규격의 핵심이다.
   - `ARCHITECTURE.md` — 사이트 뼈대(폴더·데이터·기능 레이어·확장 절차)
   - `CONTENT-SPEC.md` — 연구 문서 형식·품질·집필 원칙
   - `templates/` — 빈 템플릿(파싱·인터라이너) + 공통 CSS
   - `tools/` — 관주·이동 지도·검증 스크립트(기계라 결과가 일정)
2. **이번 작업 자료** — 대상 성경 본문(개역개정 4판 파일), 주석 자료(EPUB/PDF), 필요 시 관주 원본(`cross_references.txt`).
3. **작업 지시** — 아래 예시 문구.

## 첫 메시지 예시

### A. 새 장을 기존 책에 추가 (예: 느헤미야 2장 인터라이너)

```
느헤미야 성서 연구 서고 작업을 이어서 할게. 첨부한 zip을 풀어 구조를 확인해줘.

규격은 두 문서를 그대로 따른다:
- ARCHITECTURE.md (사이트 뼈대)
- CONTENT-SPEC.md (연구 문서 형식·표기·원칙)

이번 작업: 느헤미야 2장 인터라이너를 만든다.
- templates/interlinear-template.html 의 칸을 채우는 방식으로.
- 개역개정은 첨부한 성경 파일과 단어 단위로 대조(4판 기준).
- 주석 칩은 K/F/T/B/H, 주석 자료는 첨부본 근거.
- 문장 규칙(CONTENT-SPEC §5)과 집필 역할 경계(§0)를 지킬 것.
- 완성 후 tools/validate.py 로 검증.
중요: 히브리어 모음·형태는 실제 도구(STEP Bible 등)로 재확인 전제. 불확실하면 표시해줘.
```

### B. 새 책을 시작 (예: 창세기)

```
성서 연구 서고에 창세기를 추가할게. zip을 풀어 구조와 두 규격 문서
(ARCHITECTURE.md, CONTENT-SPEC.md)를 먼저 읽어줘.

절차는 ARCHITECTURE.md "새 책을 추가하는 법"을 따른다:
1) ot/genesis/ 에 연구 문서 작성(우선 1장 파싱부터)
2) python3 tools/build_xrefs.py cross_references.txt Gen
3) python3 tools/build_links.py ot/genesis gen
4) catalog.js 에 창세기 항목 추가
5) tools/validate.py 검증

형식은 templates/ 의 빈 템플릿을 채워서. 주석은 창세기 표준 주석
(첨부본)으로 하되, 칩 방식(약칭 한 글자+색)은 CONTENT-SPEC §3 그대로.
원어는 소프트웨어 재확인 전제.
```

### C. 원본을 안 건드리고 기능만 확장 (예: 관주를 원본 세 연구에 연결)

```
zip을 풀어줘. 원본 연구 문서는 수정하지 말고(데이터 유실 금지),
ARCHITECTURE.md "페이지에서 기능 켜기"에 따라 각 문서에
theme.css/app.css/app.js 연결과 data-book/data-chapter/data-kind/
data-root 속성, 절 요소 data-v 만 추가해줘. 그러면 관주·이동 링크가
자동으로 뜬다. 작업 후 tools/validate.py 검증.
```

## 인계 시 반드시 상기시킬 것

다른 AI는 세부 판단·문체 감각이 다를 수 있다. 특히 아래를 명시적으로 요구한다.

1. **원어 재확인.** 어느 AI든 히브리어 정확성은 편차가 크다. "원어 형태·모음은 실제 소프트웨어(Accordance/Logos/STEP Bible)로 재확인 전제, 불확실하면 표시"를 지시에 넣는다. (CONTENT-SPEC §1)
2. **개역개정 4판 단어 단위 대조.** 판본 혼입 금지. 불일치는 첨부 파일 기준 수정. (§2)
3. **집필 역할 경계.** 논지·문장·구성·은유는 사용자 창작 영역, AI는 정리·조합·검증. (§0)
4. **문장 규칙 7개.** 부정 후 긍정 금지 등. (§5)
5. **정직성.** 자료 손상(예: OCR)·기억 재구성 위험(명단 장 고유명사)을 숨기지 말고 밝힌다. (§1)
6. **데이터 유실 금지.** 원본을 융합·요약으로 덮어쓰지 않는다. 연결은 절 앵커·데이터 레이어로. (§6, ARCHITECTURE)
7. **검증 통과.** `tools/validate.py` 오류 0 확인 후 제출. (§9)

## 무엇이 자동으로 일정하게 유지되는가

- **시각(색·서체·반응형·다크모드)** → `assets/theme.css`·CSS 변수·`templates/*.css`가 상수. 모델이 바뀌어도 유지.
- **데이터 생성(관주·이동 지도)** → 스크립트가 기계적으로 처리. 결과 동일.
- **문서 골격(파싱 카드·인터라이너 7열 표)** → 빈 템플릿이 고정. 칸만 채우면 됨.
- **검증 기준** → `validate.py`가 동일 잣대로 점검.

## 무엇이 흔들릴 수 있는가 (그래서 규격서로 못박음)

- 원어 정확성(모델별 편차) → 소프트웨어 재확인으로 방어.
- 문체·논지(창작 영역) → 사용자가 직접 쓰거나 검수. AI는 자료 정리까지.
- 주석 해석의 뉘앙스 → 첨부 주석 근거로 한정, 칩으로 출처 표시.

---

## 요약

**"패키지(zip) + 자료 파일 + 주석 + 이 지시 문구"** 네 가지만 있으면, 어느 채팅·어느 AI에서든 같은 규격으로 작업을 이어갈 수 있다. 규격서 두 개(`ARCHITECTURE.md`·`CONTENT-SPEC.md`)와 템플릿·스크립트가 "항상 같아야 하는 것"을 붙잡아 주고, 자료는 그때그때 갈아 끼운다.
